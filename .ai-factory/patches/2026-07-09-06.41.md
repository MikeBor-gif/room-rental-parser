# scrape.yml: очередь GitHub Actions забивалась из-за внешнего cron-job.org

**Date:** 2026-07-09 06:41
**Files:** .github/workflows/scrape.yml
**Severity:** medium

## Problem

В GitHub посыпались ошибки на джобе `scrape`: "Internal server error"
(Correlation ID в UI Actions) и "The job was not acquired by Runner of type
hosted even after multiple attempts". Часть прогонов уходила в `cancelled`
(38% из последних 100 завершённых запусков).

## Root Cause

Проверка через `gh api repos/.../actions/runs` показала, что 100% последних
запусков триггерились событием `workflow_dispatch`, а не `schedule` — встроенный
крон `*/5 * * * *` тут почти ни при чём. Внешний сервис cron-job.org (настроен
пользователем, потому что штатное расписание GitHub Actions срабатывает с
большими и нестабильными задержками) дёргает
`POST .../actions/workflows/scrape.yml/dispatches` каждые ~2 минуты.

При этом в workflow стоял `concurrency: group: scrape, cancel-in-progress: false`
— новый dispatch не отменял предыдущий прогон, а вставал в очередь позади него.
Когда прогон подвисал (наблюдался конкретный run, ждавший раннера 15+ минут),
очередь в группе `scrape` росла быстрее, чем GitHub успевал её разгребать, и
инфраструктура Actions начинала ронять джобы с ошибкой acquisition раннера
или отвечать Internal Server Error на сам dispatch.

## Solution

- `cancel-in-progress: false` → `true` в блоке `concurrency`: новый dispatch от
  cron-job.org теперь отменяет зависший предыдущий прогон вместо того, чтобы
  вставать за ним в очередь — очередь больше не копится.
- Добавлен `timeout-minutes: 10` на джобу `scrape`: подвисший прогон падает
  быстро вместо ожидания раннера 15+ минут.

cron-job.org намеренно не трогали — пользователь полагается на него именно
из-за нестабильности штатного расписания GitHub Actions.

## Prevention

- Если на workflow_dispatch завязан внешний более частый триггер (keepalive/
  внешний cron), концентрация `cancel-in-progress: false` для этого workflow
  почти гарантированно приведёт к разрастанию очереди — по умолчанию должно
  быть `true`, если только явно не нужно гарантированно выполнить каждый
  запуск (тогда нужна другая стратегия, не concurrency-группа с очередью).
- Для джоб, вызываемых внешним расписанием, всегда ставить `timeout-minutes`,
  чтобы зависший прогон не блокировал очередь надолго.
- При диагностике "непонятных ошибок GitHub Actions" сначала смотреть
  `gh api repos/<repo>/actions/runs`, поле `event` и `triggering_actor` —
  это быстро показывает, кто реально дёргает workflow, а не полагаться на
  предположение, что сработал встроенный `schedule`.
