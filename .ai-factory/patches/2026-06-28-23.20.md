# realt.by: в выдачу попадали очень старые объявления

**Date:** 2026-06-28 23:20
**Files:** src/parsers/realt_rooms.py, tests/test_realt_parser.py
**Severity:** medium

## Problem

Парсер realt.by присылал в Telegram «очень старое» объявление. Список
запрашивается с сортировкой по дате (`?sortType=createdAt`), но в выдаче всё
равно оказывалось объявление со старой датой создания.

## Root Cause

В конвейере не было фильтра по возрасту. Единственный барьер — дедупликация по
`SeenStore` (id уже отправляли?), которая не учитывает дату. realt.by поднимает
в топ списка «бумпнутые»/премиум-объявления, у которых `createdAt` остаётся
старым, поэтому сортировки по дате недостаточно. Поле `createdAt` парсер уже
клал в `extra`, но нигде не использовал для отсева. У Kufar проблемы нет —
его JSON-API отдаёт строго свежие сверху.

## Solution

Добавил в `RealtRoomsParser` фильтр свежести: объявления старше `MAX_AGE_DAYS`
(= 3 дня) по `createdAt` отбрасываются в `parse()` с логом `[FIX]`. Помощник
`_is_too_old()` разбирает ISO 8601 с таймзоной; при пустом/битом `createdAt`
объявление НЕ отсекается (только WARN), чтобы смена формата realt не выкинула
разом всю выдачу. В `parse()`/`_is_too_old()` добавлен необязательный параметр
`now` для детерминированных тестов (в проде берётся текущее UTC-время).

## Prevention

- Для лент с премиум/бумп-размещениями нельзя полагаться только на сортировку
  сервера — добавлять явный фильтр свежести по дате создания.
- Любая логика, завязанная на «текущее время», должна принимать точку отсчёта
  (`now`) параметром, иначе тесты на статичных фикстурах становятся таймбомбой.
- При фильтрации лучше «fail-open» на неразборчивых данных (оставить + WARN),
  чем молча терять все элементы при смене формата источника.
