# План: Парсер объявлений об аренде комнат с уведомлениями в Telegram

**Дата создания:** 2026-06-27
**Режим:** full (без git-ветки — репозиторий ещё не инициализирован)
**Ветка:** —

## Настройки

- **Тесты:** да, базовые (логика парсинга и дедупликации на HTML-фикстурах)
- **Логирование:** подробное (DEBUG для разработки, через `LOG_LEVEL`)
- **Документация:** warn-only (отдельный таск с инструкцией по настройке)

## Обзор

Сервис периодически (по cron-расписанию GitHub Actions) обходит заданные сайты с
объявлениями об аренде комнат, извлекает новые объявления, сравнивает с базой
«уже виденных» и отправляет новые в Telegram. Постоянно работающий сервер не нужен:
GitHub Actions запускает скрипт каждые N минут, состояние («виденные» ID) хранится
в файле в репозитории и коммитится обратно после каждого прогона.

**Важно:** конкретные сайты/ссылки пользователь предоставит позже. Этот план создаёт
**инфраструктуру с подключаемыми парсерами** — добавление парсера под новый сайт
сводится к одному файлу в `src/parsers/`, реализующему общий интерфейс.

## Стек

- **Язык:** Python 3.14
- **HTTP/парсинг:** httpx, BeautifulSoup4 (lxml); Playwright — опционально, для сайтов с JS
- **Telegram:** прямые вызовы Bot API через httpx (без тяжёлых зависимостей)
- **Хранилище состояния:** SQLite-файл `data/seen.db` (либо JSON) в репозитории
- **Планировщик/деплой:** GitHub Actions (cron + ручной запуск)
- **Тесты:** pytest

## Архитектура (целевая структура)

```
parser/
├── .github/workflows/scrape.yml   # cron каждые N минут + ручной запуск + commit состояния
├── src/
│   ├── __init__.py
│   ├── main.py            # оркестратор: парсеры → дедуп → Telegram → сохранение
│   ├── config.py          # чтение env (токен, chat_id, LOG_LEVEL, фильтры)
│   ├── logging_setup.py   # настройка логгера по LOG_LEVEL
│   ├── models.py          # dataclass Listing (id, title, price, url, ...)
│   ├── http_client.py     # httpx-сессия: заголовки, таймауты, ретраи
│   ├── storage.py         # хранилище «виденных» ID (SQLite)
│   ├── telegram.py        # отправка сообщений в Telegram
│   └── parsers/
│       ├── __init__.py
│       ├── base.py        # BaseParser — интерфейс парсера сайта
│       └── example_site.py# парсер-шаблон (заглушка до получения ссылок)
├── tests/
│   ├── fixtures/          # сохранённые HTML-страницы для тестов
│   ├── test_storage.py
│   ├── test_dedup.py
│   └── test_example_parser.py
├── data/
│   └── .gitkeep           # сюда коммитится seen.db
├── requirements.txt
├── .env.example
├── .gitignore
├── pyproject.toml
└── README.md
```

**Поток работы (`main.py`):**
1. Загрузить конфиг и список активных парсеров.
2. У каждого парсера вызвать `fetch()` → список `Listing`.
3. Отфильтровать те, чьих `id` нет в `storage` (новые).
4. Применить пользовательские фильтры (цена, район — из конфига, опционально).
5. Отправить новые в Telegram (по одному, с защитой от rate-limit).
6. Записать новые `id` в `storage`.
7. GitHub Action коммитит обновлённый `data/seen.db` обратно в репозиторий.

### Фаза 0 — Предусловия (инфраструктура)

- **T1. Инициализировать git-репозиторий и репозиторий на GitHub**
  GitHub Actions требует репозиторий на GitHub. `git init`, первый коммит,
  создать приватный репозиторий и запушить. Файлы: `.gitignore`.
  Лог: вывести в README шаги — здесь кода нет, но это блокер для Фазы 3.

### Фаза 1 — Каркас проекта

- ✅ **T2. Скелет Python-проекта**
  Создать `pyproject.toml`, `requirements.txt` (httpx, beautifulsoup4, lxml, pytest),
  `.gitignore` (`.venv`, `__pycache__`, `.env`), `.env.example`, пустой `README.md`,
  структуру `src/` и `data/.gitkeep`. Завести виртуальное окружение и установить зависимости.
  Логирование: подтвердить установку в README/логе.
  Блокирует: T3, T4, T5, T6, T7.

- ✅ **T3. Конфигурация, логирование и модель данных**
  `src/config.py` — читает env: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `LOG_LEVEL`,
  опц. фильтры (`MAX_PRICE`, `KEYWORDS`). `src/logging_setup.py` — логгер по `LOG_LEVEL`.
  `src/models.py` — `@dataclass Listing` с обязательным стабильным `id` и методом
  форматирования для Telegram. Лог: DEBUG — какие значения конфига загружены (без секретов!).
  Зависит от: T2.

- ✅ **T4. HTTP-клиент**
  `src/http_client.py` — обёртка над httpx: общая сессия, User-Agent, таймауты,
  ретраи с backoff, обработка не-200. Лог: DEBUG — URL/статус каждого запроса, WARN — ретраи.
  Зависит от: T2.

### Фаза 2 — Ядро парсинга

- ✅ **T5. Хранилище «виденных» объявлений**
  `src/storage.py` — SQLite (`data/seen.db`): `is_seen(id)`, `mark_seen(id, meta)`,
  автосоздание схемы. Лог: DEBUG — сколько id в базе, сколько добавлено за прогон.
  Зависит от: T2.

- ✅ **T6. Базовый интерфейс парсера + шаблон**
  `src/parsers/base.py` — абстрактный `BaseParser` с методом `fetch() -> list[Listing]`
  и общими утилитами (нормализация id из URL). `src/parsers/example_site.py` — рабочий
  шаблон-заглушка, показывающий как реализовать парсер сайта (с TODO под реальные селекторы).
  Лог: DEBUG — сколько объявлений извлёк каждый парсер. Зависит от: T3, T4.

- ✅ **T7. Отправка в Telegram**
  `src/telegram.py` — отправка сообщения через Bot API (`sendMessage`), формат карточки
  объявления (заголовок, цена, ссылка), экранирование, пауза между сообщениями от rate-limit.
  Лог: DEBUG — отправка, WARN/ERROR — ошибки API Telegram. Зависит от: T3, T4.

- ✅ **T8. Оркестратор `main.py`**
  Связать всё: загрузить парсеры → собрать объявления → отфильтровать новые через storage →
  применить фильтры → отправить в Telegram → пометить виденными. Корректная обработка ошибок
  одного парсера (не падать целиком). Лог: INFO — итоги прогона (найдено/новых/отправлено).
  Зависит от: T5, T6, T7.

### Фаза 3 — Деплой (GitHub Actions)

- ✅ **T9. Workflow GitHub Actions**
  `.github/workflows/scrape.yml`: триггеры `schedule` (cron, напр. `*/15 * * * *`) и
  `workflow_dispatch`; шаги — checkout, setup-python, install, запуск `python -m src.main`,
  затем commit изменённого `data/seen.db` обратно (через `GITHUB_TOKEN`). Секреты:
  `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`. Лог: шаги CI выводят сводку прогона.
  Зависит от: T8 (и T1 — нужен GitHub-репозиторий).

### Фаза 4 — Тесты

- ✅ **T10. Тесты хранилища и дедупликации**
  `tests/test_storage.py`, `tests/test_dedup.py` — на временной БД проверить
  `is_seen/mark_seen` и что повторные объявления не отправляются. Зависит от: T5, T8.

- ✅ **T11. Тест парсера на HTML-фикстуре**
  `tests/fixtures/` — сохранённый HTML; `tests/test_example_parser.py` проверяет,

### Фаза 5 — Документация настройки

- ✅ **T12. README: настройка бота и деплоя**
